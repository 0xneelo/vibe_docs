# Section 6: Access Phasing

## 6.1 Phase A — Access-Gated Launch

Codes are required for high-sensitivity actions such as trading and listing.
This keeps launch risk bounded while infrastructure hardens.

## 6.2 Phase B — Unified Access and Referral Identity

Access and referral converge into one identity surface:

- lower user confusion,
- cleaner attribution,
- and stronger growth instrumentation.

## 6.3 Phase C — Open Participation with Optional Benefits

The long-term target removes hard code requirements for core participation.
Codes become an optional acceleration layer rather than a mandatory gate.

## 6.4 Why Phasing Matters

Phasing avoids two failure modes:

- launching fully open before abuse controls are mature,
- or staying gated too long and throttling network effects.

## 6.5 Migration Requirements

Any phase transition should include:

- backward compatibility windows,
- code format migration rules,
- and transparent communication on what changes for traders, listers, and partners.

## 6.6 Launch Sequencing Notes

Transcript planning emphasizes a staged rollout:

- an access-code beta window for controlled onboarding,
- then referral-code activation,
- then broader reward-layer expansion (on-chain transferability, packs, artifact economy).

This sequencing supports two goals at once:

- early growth through scarcity and clear onboarding logic,
- and reduced abuse risk while the accounting stack hardens.

## 6.7 Referral Code Issuance Policy

A recurring design preference in the transcript is that referral codes remain qualified, not universally issued.

Possible issuance routes:
- performance thresholds (for example, volume or sustained activity),
- curated campaign grants,
- and progressive unlock criteria over time.

Keeping issuance selective helps preserve referral signal quality and avoids immediate dilution.
